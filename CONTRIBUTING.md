# Contributing to Smalltalk

Thanks for using Smalltalk. Contributions are welcome — read this before opening a PR.

---

## Proposing a new type

The type system is intentionally lean. Each type must do one thing that no existing type covers.

To propose a new type, open an issue with:

1. **The use case** — what file or content category requires this type
2. **What prose it replaces** — show the markdown before and the Smalltalk after
3. **A proposed syntax** — one-line example following the `TYPE: id | field | field` pattern
4. **Why an existing type doesn't cover it** — specifically which types you considered and why they fall short

New types are added when the use case is genuinely distinct. Stretching an existing type to cover an edge case is preferred over adding a new one.

---

## Improving examples

Examples in `examples/` are the fastest way to help. Good contributions:

- Fix a `.st` entry that misuses the grammar
- Add a missing skill in `examples/skills/`
- Add a realistic agent workflow in `examples/agents/`
- Add a memory scenario in `examples/memory/`

All `.st` files should be written directly in Smalltalk grammar — not generated by an LLM then pasted. They are the canonical reference for what correct `.st` looks like.

---

## Improving the CLI

The CLI lives in `smalltalk/`. Each command is a separate module. To contribute:

1. Fork the repo
2. Create a branch: `feat/your-feature` or `fix/your-fix`
3. Follow the existing module pattern — one command per file
4. Add a test if behaviour is testable
5. Open a PR with a clear description of what changed and why

---

## Improving the spec

`spec/grammar.md` is the canonical type reference. Changes here affect every downstream tool and user. Changes to the spec require:

- A clear rationale
- Before/after examples
- Backwards compatibility assessment

## Code of conduct

Be direct. Be specific. Prefer working examples over abstract proposals.
